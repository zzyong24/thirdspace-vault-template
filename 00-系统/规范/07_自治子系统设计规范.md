# 自治子系统设计规范

## 目标

ThirdSpace 的每个工作区不是普通文件夹，而是一个自治子系统。自治子系统必须能被 Agent 理解、进入、维护、审计和修复。

本规范解决三个问题：

- Agent 进入任意目录后，如何知道自己处于什么工作状态。
- 每个工作区如何独立维护自己的输入、输出、状态和边界。
- 当结构漂移、标签混乱、文件滞留、内容放错位置时，如何发现并恢复。

## 子系统接口

每个子系统必须定义以下接口：

| 接口 | 说明 |
|---|---|
| 边界 | 本子系统负责什么，不负责什么 |
| 输入 | 允许进入的内容类型和来源 |
| 输出 | 处理后流向哪里 |
| 状态 | 允许出现的 `status` 值 |
| 规则 | 命名、Frontmatter、目录、标签约束 |
| Skill | 维护本子系统的主 Skill |
| 审计 | 可机器检查的健康指标 |
| 修复 | 可自动修复、半自动修复和必须人工确认的动作 |

## 标准状态

| 状态 | 含义 | 典型工作区 |
|---|---|---|
| `draft` | 草稿、未完成、待整理 | 收件箱、输出 |
| `active` | 当前有效、可继续维护 | 日记、知识、项目、资源 |
| `processed` | 已处理并完成流转 | 收件箱 |
| `ready` | 可发布或可交付 | 输出 |
| `published` | 已发布 | 输出 |
| `archived` | 已归档，不参与日常工作流 | 归档 |

## 自治维护回路

每个子系统都必须按同一条维护回路运行：

```text
识别当前工作区
  -> 读取子系统契约
  -> 接收或定位文件
  -> 判断内容类型和状态
  -> 应用命名、Frontmatter、标签规则
  -> 判断是否需要流转
  -> 写入索引、队列或报告
  -> 审计结构漂移
  -> 修复低风险问题
```

## 自动修复分级

| 等级 | 可执行动作 | 例子 |
|---|---|---|
| 自动修复 | 不改变语义、不移动跨工作区内容 | 补 `modified`、补 `workspace`、创建缺失空目录 |
| 半自动修复 | 需要写报告或 queue，但不立即移动 | 收件箱文件建议迁入知识区、资源缺少索引 |
| Trace 修复 | 会移动项目、改变归属或归档内容 | 项目跨分类移动、内容从日记迁到知识 |
| 人工确认 | 证据不足或影响活跃项目 | 低置信度项目分类、删除历史上下文 |

## 子系统列表

| 工作区 | 子系统 | 维护目标 |
|---|---|---|
| `00-系统` | System Control Plane | 保持规范、Schema、Skill、审计报告一致 |
| `01-收件箱` | Intake Router | 接收内容、粗分、生成流转队列，避免长期堆积 |
| `02-日记` | Timeline System | 保持时间线和反思复盘清晰，不沉淀长期知识 |
| `03-知识` | Knowledge Memory | 沉淀长期知识，避免原始材料和项目任务污染 |
| `04-项目` | Project Runtime | 保护活跃项目上下文，按项目意图分类推进 |
| `05-资源` | Resource Library | 管理可复用资料和附件索引，避免资源黑箱 |
| `06-输出` | Publishing Pipeline | 管理从草稿到发布的成品生命周期 |
| `99-归档` | Archive Ledger | 保存历史和迁移 trace，不参与日常工作流 |

## 流转协议

常见流转路径：

```text
01-收件箱/网页剪藏 -> 03-知识 或 05-资源
01-收件箱/临时想法 -> 02-日记 或 04-项目
02-日记/工作日志 -> 03-知识
03-知识 -> 06-输出
04-项目 -> 06-输出
04-项目 -> 99-归档
05-资源 -> 03-知识 或 04-项目
06-输出 -> 99-归档
```

跨工作区移动必须记录：

- 原路径
- 目标路径
- 判断理由
- 执行时间
- 执行者或执行 Skill
- 是否需要人工复核

## Skill 责任

工作区 Skill 不只是写文件说明，而是该子系统的维护器。每个工作区 Skill 必须包含：

- 规则来源
- 文件创建规则
- 自治维护回路
- 审计项
- 修复策略
- 禁止事项

根 Skill `thirdspace-vault` 负责：

- 根据 `pwd` 路由到子系统
- 提供初始化能力
- 提供全库结构审计
- 提供子系统契约审计
- 提供项目分类审计

## 验收标准

自治子系统落地后，至少满足：

- 每个工作区有 `WORKSPACE.md`。
- 每个工作区有对应 `workspace-*` Skill。
- `.thirdspace/schema/subsystems.yaml` 覆盖所有工作区。
- `thirdspace-vault.mjs audit-subsystems --write-report` 能生成审计报告。
- 活跃工作区无非预期深层嵌套。
- 收件箱、输出、项目、归档存在明确状态流转。
