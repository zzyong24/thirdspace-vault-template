# 项目工作区分类治理规则

## 目标

`04-项目` 按项目意图分类，而不是按历史来源分类。分类的目标是让 Agent 进入任意项目目录后，能快速判断当前项目的工作状态、应加载的 Skill、允许创建的文件类型和产出去向。

## 一级分类

| 分类 | 目录名 | 判定标准 | 典型内容 |
|---|---|---|---|
| 内容创作 | `内容创作` | 目标是生产视频、文章、课程、栏目、选题或发布包 | 视频项目、口播稿项目、栏目工程、创作流水线 |
| 产品系统 | `产品系统` | 目标是建设软件、工具、系统、插件、技术产品 | MoonOS、灵犀、Msg-Collect、知识库产品 |
| 运营增长 | `运营增长` | 目标是社群、知识星球、内容运营、增长飞轮、销售转化 | 知识星球、社群 SOP、内容矩阵、增长看板 |
| 商业合作 | `商业合作` | 目标是客户线索、代理商、报价、合作方案、商业模式验证 | 商业构想、代理商产品、客户方案 |
| 研究验证 | `研究验证` | 目标是阶段性调研、技术验证、方案探索，还没有明确交付项目 | 调研项目、技术 spike、方案评估 |
| 实验原型 | `实验原型` | 目标是 MVP、原型、未确定方向的低成本试验 | AI 童伴、早期原型、一次性验证 |

## Frontmatter 字段

项目入口文档或项目内核心文档应尽量包含：

```yaml
workspace: "04-项目"
type: "project"
project: "项目名"
project_type: "content|product|operations|business|research|experiment"
project_category: "内容创作|产品系统|运营增长|商业合作|研究验证|实验原型"
stage: "idea|research|prototype|active|paused|completed|archived"
status: "active"
```

## 分类原则

- 优先看项目目标，而不是文件来源。
- 一个项目只能有一个主分类，其他维度写入 `tags` 或 `domain`。
- 复杂项目内部结构不打散，例如 `_assets/`、`research/`、`renders/`、`.codex/skills/`。
- 低置信度分类只生成建议，不自动移动。
- 跨一级分类移动必须写入迁移 trace，包含原路径、目标路径、判断理由、时间。

## Agent 操作规则

1. 先运行项目分类审计，输出建议，不直接移动。
2. `confidence >= 0.8` 的建议可进入待执行清单。
3. `confidence < 0.8` 的建议必须人工确认。
4. 执行移动时只能移动完整项目目录，不移动项目内部零散文件。
5. 移动后更新项目入口文档、报告和 trace。
